# Isty alignment tasks (конструктор + презентация КП)

Goal: привести SCOPELIST в соответствие с методологией КП из репозитория `Isty-org/sales`, чтобы он работал и как **продающая презентация**, и как **компонентный конструктор**, а не как «таблица сметы».

Документ написан как чек-лист для агентов/разработчиков. Каждая задача самодостаточна: зачем, какие файлы, шаги, как проверить, какие риски.

## Контекст: 5 принципов Isty (источник правды)

Из `Isty-org/sales` (`HUMANS.md`, `specs/{proposal-system,content,workflow}.md`, `docs/{NEW_KP_INSTRUCTION,DESIGN_SYSTEM,PROPOSAL_COMPONENTS}.md`):

1. **Коммерческий каркас:** обещание → быстрые факты → боль «почему сейчас» → целевой контур → роли/сценарии/модули/варианты → состав и этапность запуска + границы → прозрачные деньги → доверие (почему мы, контроль человека) → конкретный следующий шаг.
2. **Конструктор из блоков:** общие компоненты + уникальные рядом со страницей, **данные отделены от шаблона**, страница = композиция блоков, не монолит.
3. **Архетипы сделок** определяют, какие блоки усилить (операционный контур / стратегический проект / сравнение вариантов / пакет ассистентов / быстрый пакетный запуск / модульная смета).
4. **Дизайн-система и тон:** токены `main #020B14`, `accent #8E44AD`, `stepM`, `stepA #E67E22`, шрифт Onest, glass-карточки, glow-блобы, строка микро-доверия; тон рациональный/дерзкий/комфортный, проверяемые формулировки, без хайпа.
5. **Контентная честность:** факты только из документов; неопределённость как допущение/вопрос; не выдумывать scope/интеграции/SLA/цифры; не обещать автономный ИИ без человека.

**Ключевая находка аудита (2026-06-08):** богатые коммерческие поля уже объявлены в `src/lib/types.ts:93` (`clientProblem`, `businessGoal`, `proposedSolutionSummary`, `whyUs`, `processSteps`, `proofItems`, `packages`, `nextStepText`), но живут только в серверной модели `Proposal`/сторе. Ни билдер (`ProjectSettingsForm.tsx`), ни презентация (`ProposalPreview.tsx`) их не используют. Большая часть нужного уже есть в данных и просто не выведена наружу.

Рекомендуемый порядок (максимум эффекта при минимуме риска): **T1 → T2 → T6 → T3 → T4 → T5**.

## T1. Подключить существующие коммерческие поля к билдеру и рендеру

**Зачем.** Самый дешёвый и самый крупный разрыв. Поля `clientProblem`, `businessGoal`, `proposedSolutionSummary`, `whyUs`, `processSteps`, `proofItems`, `nextStepText` уже есть в модели, но не редактируются и не рендерятся. Их вывод сразу превращает смету в полноценный коммерческий нарратив (принцип 1).

**Файлы:** `src/lib/types.ts` (модель билдера `ProjectSettings`/`ProposalData`), `src/components/ProjectSettingsForm.tsx`, `src/components/ProposalPreview.tsx`, при необходимости `src/lib/server/proposal-store.ts` (маппинг).

**Шаги:**
1. Решить вопрос двух моделей: сейчас билдер использует тонкую `ProjectSettings`/`ChangeItem`, а серверная `Proposal` — богатая. Свести: либо расширить `ProjectSettings` нужными полями, либо начать редактировать `Proposal` напрямую. Зафиксировать выбор в PR.
2. Добавить в форму билдера разделы: «Контекст и боль» (`clientContext`, `clientProblem`), «Цель и решение» (`businessGoal`, `proposedSolutionSummary`), «Почему мы» (`whyUs`), «Следующий шаг» (`nextStepText`).
3. Добавить редактируемые списки «Процесс/этапы» (`processSteps`) и «Кейсы/доказательства» (`proofItems`).
4. В `ProposalPreview` отрендерить секции в порядке коммерческого каркаса (см. принцип 1).
5. Прогнать `npm run lint && npm run build`.

**Acceptance:**
- Все перечисленные поля редактируются в билдере и видны в публичном виде `/p/<shareSlug>`.
- Порядок секций соответствует коммерческой дуге.
- `npm run build` зелёный.

**Риски:** расхождение моделей `ProjectSettings` ↔ `Proposal` может потребовать миграции данных в сторе/Supabase. Проверить обратную совместимость уже опубликованных записей.

## T2. Заменить авто-«ключевые выводы» на смысловой Hero + микро-доверие

**Зачем.** Сейчас «Ключевые выводы» (`ProposalPreview.tsx:55`) — механические строки («3 обязательных изменений включены…»). По тону Isty это анти-паттерн: счётчики вместо смысла. Нужен первый экран с обещанием (что делаем + зачем) и строка микро-доверия (принципы 1 и 4).

**Файлы:** `src/components/ProposalPreview.tsx`, `src/components/PublicProposalView.tsx`.

**Шаги:**
1. Убрать массив `highlights` с авто-генерацией или свести к фактической сводке (бюджет/сроки) без выдаваемых за «выводы» счётчиков.
2. Сделать Hero: метка → H1 (обещание) → подзаголовок (польза) → строка микро-доверия (например `Фиксируем объём • Прозрачные сроки • Без скрытых работ`, формулировка под SCOPELIST).
3. Следующий шаг брать из `nextStepText` вместо захардкоженного абзаца в `PublicProposalView.tsx:111`.

**Acceptance:**
- На первом экране есть смысловое обещание и строка микро-доверия.
- Текст «следующего шага» приходит из данных, а не зашит в код.

**Риски:** формулировки должны оставаться проверяемыми — не вводить обещаний, которых нет в данных.

## T3. Разбить монолитную презентацию на переиспользуемые блоки

**Зачем.** `ProposalPreview` — один монолит с фиксированным набором и порядком секций; каждое КП выглядит одинаково. Isty прямо запрещает монолит и требует библиотеку блоков, включаемых под задачу (принцип 2).

**Файлы:** новый каталог `src/components/proposals/**`, рефактор `src/components/ProposalPreview.tsx`.

**Шаги:**
1. Вынести секции в блоки: `Hero`, `Snapshot`, `Problem`, `Solution`, `Scope`, `Pricing`, `Timeline`, `Proof`, `WhyUs`, `Process`, `FinalCta`.
2. `ProposalPreview` превратить в композицию блоков по конфигу.
3. Управлять видимостью через уже существующие флаги `ShareSettings` (`showPrices`, `showTimeline`, `showComparisonTable` в `src/lib/types.ts:30`) + добавить недостающие.
4. Рефактор без визуальной регрессии (правило Isty `workflow.md`): внешний вид до и после совпадает.

**Acceptance:**
- Каждый блок — отдельный файл, страница их импортирует.
- Флаги `ShareSettings` реально скрывают/показывают блоки.
- Визуальной регрессии нет (сверить скриншоты до/после).

**Риски:** легко словить визуальный регресс при выносе — делать поэтапно, по одному блоку.

## T4. Тема и токены дизайн-системы Isty для клиентского вида

**Зачем.** Клиентская презентация сейчас zinc/emerald/белая бумага, цвета захардкожены в каждом блоке. Isty требует тёмную палитру и только токены (принцип 4). Примечание: фирменный фиолетово-оранжевый градиент уже есть в *билдере* (см. историю коммитов), но не доехал до клиентского КП.

**Файлы:** `src/app/globals.css`, блоки презентации после T3.

**Шаги:**
1. Завести CSS-токены: `--main #020B14`, `--accent #8E44AD`, `--stepM`, `--stepA #E67E22`; подключить Onest (локально, как уже сделано с Geist — см. `README.md`).
2. Заменить захардкоженные `text-emerald-*`/`bg-zinc-*` на токены.
3. Добавить glass-карточки и мягкий glow как опциональную тему «Isty»; согласовать со светлой/тёмной темой (`ThemeToggle`).

**Acceptance:**
- Цвета берутся из токенов, не подбираются вручную.
- Доступна фирменная тема Isty в клиентском виде.

**Риски:** контраст и читаемость на тёмном фоне (в истории уже были правки pink-on-purple) — проверить контраст ключевых блоков.
